# H200 V2 实验运行手册

本文只适用于 v2 项目。代码、运行脚本、prediction、metrics 和新建 ES index 都必须落在 v2 项目下。旧项目只允许作为只读离线产物来源，用来复用已经存在的数据集、baseline cache 或 ES index。

```text
本地 v2 项目：/home/ruolinsu/signpost/signpost_re_v2
H200 v2 项目：/home/srl/signpost_re_v2
H200 旧项目：/home/srl/signpost_re  # 只读复用离线产物，不写入
```

## 3. 每个数据集一条命令

每个 tmux 内部按顺序执行：

```text
service smoke check
baseline-v2 独立 chunk ES index
Signpost full + ablations
vanilla_llm
hybrid_rag
cluerag
cluerag_prompt_normalized
agrag
linearrag
hiprag
graphrag_r1
target/silver v2 metrics
final v2 metrics
```

每个方法内部按 question 并发执行，默认：

```bash
V2_QUERY_WORKERS=8
SIGNPOST_QUERY_WORKERS=8
BASELINE_QUERY_WORKERS=8
```

如 8000 chat 服务稳定，可以把 `V2_QUERY_WORKERS` 提高到 12 或 16；如果 8001 embedding 服务不稳定，先保持 8。并发只发生在同一方法内部的 query 级别，prediction 和 query log 会按 question 顺序统一写回，不会多个 worker 同时 append 同一个 JSONL。

直接启动：

```bash
tmux new-session -d -s v2-agriculture "bash -lc '/home/srl/signpost_re_v2/scripts/h200/run_v2_dataset_all.sh agriculture agriculture agriculture'"
tmux new-session -d -s v2-mixv0 "bash -lc '/home/srl/signpost_re_v2/scripts/h200/run_v2_dataset_all.sh mixv0 mix mix'"
```

参数含义：

```text
第 1 个参数：输出 dataset，写 outputs/<dataset>
第 2 个参数：namespace
第 3 个参数：processed dataset，读 datasets/processed/<processed>
```

因此 `mixv0` 会写：

```text
/home/srl/signpost_re_v2/outputs/mixv0/
```

并读取：

```text
/home/srl/signpost_re_v2/datasets/processed/mix/
```

`mixv0` 的 namespace 必须用 `mix`，这样 Signpost/ClueRAG 才能复用旧 ES 中已经存在的 `signpost-mix-*` 和 `cluerag-mix-multilayer`。输出 dataset 仍是 `mixv0`，不会写到旧项目。

## 5. 复用重跑

Hybrid RAG 必须构建 v2 自己的独立 ES chunk index：

```text
baseline-v2-<dataset>-chunks
```

其他离线产物可以复用。文件型 cache 可以复制到 v2，也可以通过只读源路径让 v2 代码读取；新的 predictions/logs/metrics 仍写在 `/home/srl/signpost_re_v2/outputs/<dataset>/`。ES 型 cache 通过显式 index name 复用，不复制文件。

文件型复用对象：

```text
datasets/processed/<processed_dataset>/*.jsonl
datasets/processed/<processed_dataset>/graph.*.json
outputs/<dataset>/baselines/agrag/index.pkl
outputs/<dataset>/baselines/linearrag/index.pkl
outputs/<dataset>/baselines/hiprag/index.pkl
outputs/<dataset>/baselines/graphrag_r1/index.pkl
outputs/<dataset>/baselines/cluerag/shared_graph/
```

首次 v2 正式跑时，Hybrid RAG 的 `baseline-v2-<dataset>-chunks` 还不存在，所以不要设置 `REUSE_BASELINE_CHUNK_INDEX=1`。可以复用旧 processed 数据和 ClueRAG graph：

```bash
tmux new-session -d -s v2-agriculture "bash -lc 'V2_PROCESSED_SOURCE_DIR=/home/srl/signpost_re/datasets/processed/agriculture V2_BASELINE_SOURCE_DIR=/home/srl/signpost_re/outputs/agriculture/baselines REUSE_GRAPH=1 REUSE_CLUERAG_OUTPUTS=0 V2_QUERY_WORKERS=12 /home/srl/signpost_re_v2/scripts/h200/run_v2_dataset_all.sh agriculture agriculture agriculture'"
tmux new-session -d -s v2-mixv0 "bash -lc 'V2_PROCESSED_SOURCE_DIR=/home/srl/signpost_re/datasets/processed/mix V2_BASELINE_SOURCE_DIR=/home/srl/signpost_re/outputs/mix/baselines REUSE_GRAPH=1 REUSE_AGRAG_INDEX=1 REUSE_LINEARRAG_INDEX=1 REUSE_HIPRAG_INDEX=1 REUSE_GRAPHRAG_R1_INDEX=1 REUSE_CLUERAG_OUTPUTS=0 V2_QUERY_WORKERS=12 /home/srl/signpost_re_v2/scripts/h200/run_v2_dataset_all.sh mixv0 mix mix'"
```

如果已经在 v2 ES 中建过 Hybrid 的独立 chunk index，重跑时再加：

```text
REUSE_BASELINE_CHUNK_INDEX=1
```

如果只想复用旧 ClueRAG retrieval_results 生成 prompt-normalized 版本，可显式加：

```text
REUSE_CLUERAG_OUTPUTS=1
```

ES 型复用：

```text
Hybrid RAG：正式 v2 仍使用 baseline-v2-<dataset>-chunks；如果该 index 已经在 ES 里存在，设置 REUSE_BASELINE_CHUNK_INDEX=1 跳过重建。
ClueRAG：设置 REUSE_GRAPH=1 或 REUSE_BASELINE_INDEX=1，会复用 shared_graph 文件和 baseline-owned ClueRAG ES index；如果 ES index 不存在，脚本会失败。
Signpost 主流程：v2 Signpost 可读取 v2 namespace 对应 ES index；不要让 Hybrid RAG 使用 Signpost chunk index。
```

实现细节：

```text
AGRAG / LinearRAG / HiPRAG / GraphRAG-R1：
  V2_BASELINE_SOURCE_DIR 只作为 --reuse-index-dir 读取旧 index.pkl；
  graph.json / retrieval_index.json / run_metrics.json 写入 v2 outputs。

ClueRAG：
  V2_BASELINE_SOURCE_DIR 下的 cluerag/shared_graph 会被软链接到 v2 outputs 作为只读 graph cache；
  shared_outputs 不软链接到 v2 outputs，避免写回旧项目；
  REUSE_CLUERAG_OUTPUTS=1 时，prompt-normalized generation 只读取旧 shared_outputs/COSINE_1.00；
  正式 shared_outputs、generation 和 run_metrics 写入 v2 对应 method 目录。
```

如果 v2 目录已经有 cache，可以用复用模式只重跑 online query/final generation：

```bash
tmux new-session -d -s v2-agriculture-rerun "bash -lc 'REUSE_BASELINE_CHUNK_INDEX=1 REUSE_GRAPH=1 /home/srl/signpost_re_v2/scripts/h200/run_v2_dataset_all.sh agriculture agriculture agriculture'"
tmux new-session -d -s v2-mixv0-rerun "bash -lc 'REUSE_BASELINE_CHUNK_INDEX=1 REUSE_GRAPH=1 REUSE_AGRAG_INDEX=1 REUSE_LINEARRAG_INDEX=1 REUSE_HIPRAG_INDEX=1 REUSE_GRAPHRAG_R1_INDEX=1 /home/srl/signpost_re_v2/scripts/h200/run_v2_dataset_all.sh mixv0 mix mix'"
```

复用模式不会重建 `baseline-v2-<dataset>-chunks` ES index；AGRAG、LinearRAG、HiPRAG、GraphRAG-R1 只有在各自 `REUSE_*_INDEX=1` 时才复用 `index.pkl`。如果这些 cache 没有放在 v2 输出目录或通过 `V2_BASELINE_SOURCE_DIR` 暴露给 v2，脚本会失败，不会静默重建。

## 6. 并发策略

推荐先并发跑两个数据集级 tmux：

```text
v2-agriculture
v2-mixv0
```

单个数据集内部的方法顺序保持不变，因为同一数据集的 `outputs/<dataset>/metrics/*.json` 会被多个方法持续汇总更新；但每个方法内部的 query 默认 8 并发，可以通过 `V2_QUERY_WORKERS` 同时控制 Signpost 和 baseline，也可以分别设置 `SIGNPOST_QUERY_WORKERS`、`BASELINE_QUERY_WORKERS`。

实现方式是 query 级线程池并发请求模型/检索服务，完成后按原始 question 顺序统一写 prediction JSONL 和 query log。因此数据集内部能并发吃满 H200 服务，但不会多个 worker 同时 append 同一个输出文件。

建议：

```text
稳定起跑：V2_QUERY_WORKERS=8
8000 chat 服务稳定后：V2_QUERY_WORKERS=12
8000/8001/8033 都稳定后：V2_QUERY_WORKERS=16
```

## 8. V2 指标口径

所有方法计算 TargetUnitRecall。Silver evidence 指标只计算：

```text
signpost.full
hiprag
graphrag_r1
```

Silver 指标只读取 prediction 的 `evidence_chunks`，不 fallback 到 `citations` 或 `retrieved_chunks`。单轮方法保留 `retrieved_chunks` 供人工检查，但 Silver 指标填 NA。

指标输出：

```text
outputs/<dataset>/metrics/target_unit_silver_eval_v2/method_target_unit_silver_metrics.tsv
outputs/<dataset>/metrics/target_unit_silver_eval_v2/per_query_target_unit_silver_metrics.tsv
outputs/<dataset>/metrics/final_eval_v2/method_final_metrics.tsv
outputs/<dataset>/metrics/final_eval_v2/per_query_final_metrics.tsv
```
